## 本日完了したステップ

### Step 7 — 通知キャンセル同期（AlarmManager との整合性）
- `NotificationScheduler` インターフェースを `cancel(item: NotificationItem)` → `cancel(id: String)` に変更
- `AlarmNotificationScheduler.cancel(id)`: 新しく `buildCancelPendingIntent(id)` を追加し、`alarmManager.cancel(pi) + pi.cancel()` を呼ぶ。try-catch で包みクラッシュ防止
- **要点**: `Intent.filterEquals` は extras を比較しない。なので cancel 側は extras 無しで Intent を組んでも、schedule 時と同じ `requestCode = id.hashCode()` + 同じ component（`NotificationAlarmReceiver::class.java`）ならマッチする。title や id を extras に詰めなくて OK
- `CancelNotificationUseCase`: `repository.deleteNotification(id)` と `scheduler.cancel(id)` を順に呼ぶ。**両方を try-catch で個別に包む** — 片方が失敗してももう片方を必ず実行（JSON と Alarm の不整合を最小化）
- `DeleteNotificationUseCase` は削除（`CancelNotificationUseCase` に吸収され完全にデッドコード化したため）
- `TimeViewModel.delete(id)` は `cancelNotificationUseCase` を呼ぶ形に差し替え

- `ContactsViewModel` を 3 UseCase 構成に拡張: Get / Add / Delete

### Step 9 — Settings / Import-Export（SAF）
- `JsonDataSource.writeRaw(rawJson: String)` を追加。**`save()` を `writeRaw()` を使う形に再構築**: `save(data)` → `encodeToString` → `writeRaw()`。これで `writeRaw` がデッドコードにならず、生 JSON 書き出しの低レベルプリミティブとして機能
- `SilentRepository`: `getRaw(): DataContainer` と `replaceAll(data)` を追加（既存の `updateAll` を置換削除）
- `ExportDataUseCase`: `repository.getRaw()` → `Json { prettyPrint = true }.encodeToString()`。例外時は空の `DataContainer` をエンコードしてフォールバック（claude.md §9）
- `ImportDataUseCase`: **先に decode して検証** → 成功時のみ `replaceAll`。decode で throw したら既存データは一切壊さず無視
- `SettingsViewModel`: `StateFlow` 不要の一発イベント型。`export(onReady: (String) -> Unit)` は **コールバック形**（UseCase が suspend のため `fun export(): String` にはできない）、`import(raw: String)` はファイア&フォーゲット
- `SettingsScreen`: SAF ランチャー2種
  - `ActivityResultContracts.CreateDocument("application/json")` → `contentResolver.openOutputStream(uri).use { write(bytes) }`
  - `ActivityResultContracts.OpenDocument()` + `arrayOf("application/json")` → `openInputStream.use { readBytes().toString(UTF_8) }`
- **Export 経路の順序**: ボタン押下 → `viewModel.export { json → pendingExportJson = json; launcher.launch(filename) }`。**先に JSON を生成**してから SAF を起動する。逆順にすると SAF コールバックが非 composable スコープから suspend を呼べない
- デフォルトファイル名: `"silenthub_backup.json"`

## 重要な決定事項（Step 7〜9 追加分）

- **JSON の decode 失敗は「既存データ温存」**: Import で壊れた JSON を食わせても `replaceAll` は呼ばれず、現状データはそのまま。ユーザーが誤って不正ファイルを掴んでも損失ゼロ
- **`CancelNotificationUseCase` は try-catch を 2 つに分ける**: Repository 削除と Scheduler キャンセルは **独立に** 実行されるべき。まとめて 1 つの try で囲むと、片方の失敗でもう片方がスキップされて不整合が生まれる
- **`writeRaw` を `save` の土台にする**: ユーザー仕様では別メソッドだったが、`save` を `writeRaw` に書き換えることで「片方がデッドコード」を回避。将来 import 経路で生 JSON 直書きが必要になったとき `writeRaw` だけ呼べばよい
- **SAF の `Uri` は UI 層で `String` に変換**: `Uri` を UseCase へ渡すと domain が Android framework に依存してしまう。**ストリーム→String まで UI 層、decode 以降は UseCase** という層分けで Clean Architecture を維持
- **Export は「先に JSON 生成 → SAF 起動 → Uri 返却で書き込み」の順**: `pendingExportJson` を `remember { mutableStateOf<String?>(null) }` で保持する必要がある。SAF コールバック内で suspend UseCase を呼ぶ道はない（Compose の非 composable スコープ）

## 既知の未検証事項

- 実機/エミュレータでの動作確認はまだゼロ回
