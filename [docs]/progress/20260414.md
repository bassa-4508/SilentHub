## 本日完了したステップ

### Step 4 — ViewModel + 手動 DI
- `AppContainer`（`di/`）を新設し、`JsonDataSource` → `SilentRepository` → 各 UseCase を **アプリ起動時 1 回だけ** 生成して保持
- **重要**: Repository を ViewModel ごとに new する実装を禁止。`MainActivity.onCreate` で `AppContainer` を 1 回だけ生成し、全 Factory に同じ参照を配る
- `ContactsViewModel` / `TimeViewModel` を追加。`init { ... }` で UseCase 経由ロード、`StateFlow<UiState>` 公開
- `ViewModelProvider.Factory` を自作（Hilt/Koin/Dagger 禁止 — claude.md §12）
- UI は `viewModel(factory = factory)` と `collectAsState()` で購読。Loading / Empty / List の 3 分岐のみ

### Step 5 — 通知機能（AlarmManager exact + BroadcastReceiver + FullScreenActivity）
- `domain/scheduler/NotificationScheduler` インターフェース（ドメイン層に置くことで UseCase から Android framework 依存を切る）
- `notification/AlarmNotificationScheduler` が実装。`AlarmManager.setAlarmClock(AlarmClockInfo, pi)` を使用（Doze 貫通の最強 exact 経路）
- API 31+ の `canScheduleExactAlarms()` チェック + `SecurityException` フォールバック（claude.md §9）
- `NotificationAlarmReceiver`（BroadcastReceiver）が `NotificationCompat` を post。`setFullScreenIntent(..., true)` + `setSilent(true)` + `CATEGORY_ALARM`
- **直接 `context.startActivity()` ではなく NotificationCompat 経由**: API 29+ のバックグラウンドアクティビティ起動制限を OS 側の full-screen intent 昇格で突破する正規ルート
- `FullScreenNotificationActivity`: API 27+ は `setShowWhenLocked(true)` / `setTurnScreenOn(true)` / `KeyguardManager.requestDismissKeyguard`、API 26 は `WindowManager.LayoutParams` フラグフォールバック
- `NotificationChannels`: `IMPORTANCE_HIGH` + `setSound(null, null)` + `enableVibration(false)` + `enableLights(false)` の「無音ハイ優先度」チャネル
- `PendingIntent.FLAG_IMMUTABLE`（API 31+ 必須）を全経路で指定
- `ScheduleNotificationUseCase` は scheduler 呼び出しのみ（UseCase は純 Kotlin、Android 依存はインターフェース境界で切れている）
- `AndroidManifest`: `SCHEDULE_EXACT_ALARM` / `USE_EXACT_ALARM` / `USE_FULL_SCREEN_INTENT` / `POST_NOTIFICATIONS` の 4 パーミッション追加、Activity + Receiver 登録
- `MainActivity.onCreate` で `NotificationChannels.ensureCreated()` を `AppContainer` 生成前に呼ぶ

### Step 6 — 通知の永続化 + 一覧 UI + 削除
- `SilentRepository.deleteNotification(id)` を `filterNot` で追加
- `DeleteNotificationUseCase` を新設（`suspend operator invoke` + `withContext(Dispatchers.IO)`）
- `TimeViewModel` を 4 UseCase 構成に変更: Get / Add / Delete / Schedule
- **重要な修正**: `scheduleTestFiveSeconds()` は「schedule のみ」ではなく **add → schedule → reload** の順。そうしないと「再起動で消えるゴミ通知」になる
- `delete(id)` も `delete → reload` で即 UI 反映
- `TimeScreen` に `LazyColumn` + `Card`（タイトル / 時刻 / Delete ボタン）を配置。空状態は中央寄せの "No scheduled notifications"
- 時刻表示は `SimpleDateFormat("yyyy-MM-dd HH:mm:ss")`（Step 7 以降で設定画面からフォーマット可変にする余地あり）
- 色は全て `MaterialTheme.colorScheme.*` 参照。通知画面のみ `NotificationBackground` / `NotificationForeground` を `Color.kt` にトークン追加（design.md「黒背景 + 白テキスト」要件 + claude.md §5「直接 Color 禁止」両立のため）

## 次にやること（Step 7 候補 — どちらかを選択）

### A. 通知キャンセル同期（AlarmManager cancel）
- 削除時に `AlarmNotificationScheduler.cancel(item)` も呼ぶ
- `buildPendingIntent` は同じ hashCode で構築されるため、保存時と同じ `NotificationItem` を渡す必要あり → 削除パスでは **削除前に item を取得** してから Repository 削除
- 現状: UseCase で item の lookup が無い。`GetNotificationByIdUseCase` が必要か、`DeleteNotificationUseCase` 内で `load → filter → cancel → save` にまとめるかの判断が必要

### B. Contacts 機能（CRUD）
- Contact のタイプ（LINE / X / Instagram / Email / Phone）別 URI 起動
- まずは追加ダイアログ + 一覧 + 削除のみ
- 起動は Step 8 送りでも可

**推奨は A**: B を先にやると Time 画面に未解決の「Alarm と JSON の同期漏れ」が残り続けるため。claude.md §17「1. 通知の確実性 > 2. データの安全性 > 3. UI の一貫性」に従う。

## 重要な決定事項（Step 4〜6 追加分）

- **DI は `AppContainer` 1 箇所集約**: ユーザー指示「Repository を毎回 new しない」の徹底。全 UseCase が同じ Repository インスタンスを共有することで、将来 Repository に StateFlow/キャッシュを載せても整合する
- **`NotificationScheduler` インターフェースは domain 層に置く**: UseCase が Android framework に直接依存すると Clean Architecture の層境界を壊す。実装 `AlarmNotificationScheduler` のみ `notification/`（framework 層）に配置
- **通知は full-screen intent 経由で OS に Activity を起動させる**: Receiver から `startActivity` を直叩きしない。API 29+ の制約 + ロック画面越え両方を満たすのは NotificationCompat + `setFullScreenIntent` 経路だけ
- **scheduleTestFiveSeconds は必ず保存込み**: schedule 単独呼び出しは禁止パターン。UI の操作は必ず「Repository 保存 → Scheduler 登録 → reload」の 3 点セット
- **通知画面のテーマトークン**: `NotificationBackground = 0xFF000000` / `NotificationForeground = 0xFFFFFFFF` を `Color.kt` に追加。これは MaterialTheme の `background/onBackground` とは別に「ロック画面越しの視認性」を最優先した固定値。通常画面の色トークンとは意図的に分離
- **時刻フォーマットは `SimpleDateFormat`**: `java.time` を使ってもいいが API 26 minSdk で追加依存不要、claude.md §12「不要なライブラリ追加禁止」優先
- **`SilentRepository.updateAll` は残置**: まだ呼び元は無いが、後段の import/export 機能で使う想定

- **実機/エミュレータでの通知動作未確認**: exact alarm 権限ダイアログ・ロック画面越え・無音動作はすべて未テスト

- **`canScheduleExactAlarms()` が false のときの UI フィードバック無し**: 現状は黙って return。設定画面で誘導するのが筋
- **Alarm と JSON の同期漏れ**: 上記 Step 7-A の通り。削除時に Alarm が残る
